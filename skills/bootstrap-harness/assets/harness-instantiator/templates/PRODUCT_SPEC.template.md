# PRODUCT_SPEC

日期：`[YYYY-MM-DD]`

这是一份项目输入文档。

它负责描述：

- 这个项目要做什么
- 当前阶段要交付什么
- 什么属于范围内，什么属于非目标

它不负责描述 harness 如何实例化。那部分交给 `HARNESS_BOOTSTRAP.md`。

## 1. 项目标识

- 项目名称：
- 仓库名称：
- 项目类型：`backend` / `agent-runtime` / `web-app` / `worker-api` / `other`

## 2. 问题定义

回答这些问题：

- 这个项目要解决什么问题
- 为什么现在要做
- 如果不做，会出现什么问题

## 3. 目标用户或使用者

- 主要用户是谁
- 次要使用者是谁
- 他们最关心的结果是什么

## 4. 当前阶段目标

明确当前阶段或当前里程碑的目标。

例如：

- 证明一个最小 runtime 闭环可运行
- 交付一个最小 FastAPI 服务骨架
- 建立第一条业务闭环

## 5. 范围内功能

列出当前阶段必须交付的功能或能力。

建议控制在 3 到 7 条。

## 6. 非目标

明确当前阶段不做什么。

例如：

- 不做多 agent
- 不做复杂前端
- 不做长期记忆
- 不做分布式调度

## 7. 功能性要求

列出当前阶段必须满足的功能要求。

## 8. 非功能性要求

列出当前阶段必须满足的系统质量要求。

例如：

- 显式 contract
- bounded execution
- traceability
- async-first
- deterministic tests first

## 9. 技术约束

写出当前已知的技术决策或限制。

例如：

- Python >= 3.11
- FastAPI
- Pydantic v2
- SQLite first

## 10. 成功标准

怎么判断当前阶段成功。

最好写成可观察结果，例如：

- 某个接口可工作
- 某条执行路径可通过
- 某组测试通过
- 某个 contract 已固定

## 11. 开放问题

列出尚未确定、但后续可能影响架构或 bootstrap 的问题。

## 12. 补充背景

如有必要，补充：

- 参考项目
- 设计来源
- 外部资料
- 相关线程结论
